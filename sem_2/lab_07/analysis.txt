Сравнительный анализ многопоточного сервера и epoll-сервера

=== Потоки (TASK) ===

Multithread: 5 потоков (main + по одному на каждого из 4 клиентов)
Epoll:       1 поток (единый событийный цикл)

Многопоточный сервер создаёт отдельный поток на каждое входящее соединение.
Epoll обрабатывает все соединения в одном потоке через epoll_wait.

=== Виртуальная память (MAPS / STAT) ===

  vsize (виртуальная):  Multithread ~4 ГБ   Epoll ~2 МБ
  rss   (реально RAM):  Multithread ~1.8 МБ  Epoll ~1.1 МБ

4 ГБ виртуальной памяти у multithread — не утечка, а стеки потоков.
Каждый поток получает два региона:

  fffe10000000-fffe10021000  rw-p  pages=33    <- стек потока (132 КБ)
  fffe10021000-fffe14000000  ---p  pages=16351 <- guard region (64 МБ виртуальных)

Guard region — ловушка переполнения стека: при выходе за границу сразу
segfault вместо тихой порчи данных. Физической памяти не занимает (present=0).

=== Физическое состояние памяти (PAGEMAP) ===

Ядро выделяет страницы лениво — только при первом обращении.
Пример из multithread:

  стек потока (33 страницы) — present=2-4, остальные ещё не тронуты
  код libc.so (398 страниц) — present=314, file=314 (разделяется между всеми процессами)
  guard region (16351 стр.) — present=0 (чисто виртуальное резервирование)

У epoll карта памяти значительно чище: нет блоков под стеки потоков,
меньше регионов, меньше page fault'ов (minflt: multithread=2038, epoll=126).

=== Файловые дескрипторы (FD) ===

Multithread:
  fd 3       — слушающий сокет
  fd 6,8,11,14 — сокеты 4 клиентов (открыты одновременно)

Epoll:
  fd 3 — слушающий сокет
  fd 4 — лог-файл select_server.txt
  fd 5 — anon_inode:[eventpoll] (сам epoll-дескриптор)

У epoll нет постоянно открытых клиентских сокетов — соединение закрывается
после обработки запроса. fd 5 — специальный объект ядра, через который
epoll следит за событиями на всех остальных дескрипторах.

=== Системные вызовы (IO) ===

  syscr (read):   Multithread=3167   Epoll=6
  syscw (write):  Multithread=5283   Epoll=22

Multithread делает в 200+ раз больше системных вызовов: каждый поток
читает и пишет независимо, плюс накладные расходы на переключение контекста.
Epoll обрабатывает все события через один epoll_wait без лишних переключений.

=== Анонимные отображения ===

Анонимное отображение — регион виртуальной памяти, не привязанный к файлу.
Признак в /proc/[pid]/maps: устройство 00:00, inode 0, имя отсутствует.

  aaab00187000-aaab001a8000 rw-p 00000000 00:00 0   [heap]   <- анонимное
  ffffb3840000-ffffb39ce000 r-xp 00000000 00:41 ... libc.so  <- файловое

Создаётся через mmap(MAP_ANONYMOUS) или malloc. В pagemap у анонимных
страниц file=0 — они приватны для процесса и не разделяются с другими.

В proc_multithread.txt анонимных маппингов особенно много:
  [heap]  — куча (malloc)
  [stack] — стек главного потока
  ~40 пар rw-p / ---p — стеки и guard region'ы каждого потока

В proc_epoll.txt только heap, stack и буфер libc — поток один,
стеков под дополнительные потоки нет.

=== Анализ chdir (обход дерева каталогов) ===

Программа рекурсивно обходит дерево каталогов через opendir/readdir/chdir.

state = R (Running) — процесс активно работает, в отличие от серверов (S).
num_threads = 1 — однопоточный, нет параллелизма.
vsize = ~2.3 МБ — сопоставимо с epoll сервером.

CWD меняется по ходу работы — в момент снятия:
  /sys/kernel/slab/:0000088
Это прямое следствие вызовов chdir() при обходе: /proc/[pid]/cwd
всегда показывает текущий каталог процесса в реальном времени.

FD показывает вложенную цепочку открытых DIR-дескрипторов:
  fd 3 -> /
  fd 4 -> /sys
  fd 5 -> /sys/kernel
  fd 6 -> /sys/kernel/slab
  fd 7 -> /sys/kernel/slab/:0000408
Каждый уровень рекурсии держит открытым свой DIR — это глубина
текущего стека рекурсии в момент снятия снимка.

heap: 58 страниц (present=25) — больше чем у серверов. Программа
использует path_alloc/realloc для динамического хранения пути fullpath,
который растёт с глубиной обхода.

IO:
  wchar = 270336 байт — много вывода (имена файлов в stdout/dev/null)
  syscr = 6, syscw = 66 — мало системных вызовов чтения, много записи

=== Вывод ===

Epoll эффективнее при большом числе соединений: меньше памяти (нет стеков
потоков), меньше системных вызовов, меньше page fault'ов. Многопоточный
подход проще в реализации, но плохо масштабируется — каждый новый клиент
стоит ~4 ГБ виртуального адресного пространства и дополнительных накладных
расходов на синхронизацию потоков.
